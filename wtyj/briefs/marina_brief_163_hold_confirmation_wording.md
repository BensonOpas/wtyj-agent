# BRIEF 163 — Fix premature "Booking confirmed" wording (dashboard + Marina's reply)
**Status:** Draft | **Files:** social_agent.py, marina_agent.py, Messages.tsx (frontend), test_163 (new) | **Depends on:** — | **Blocks:** —

## Context

Image #74 from Benson on 2026-04-08 shows a WhatsApp booking confirmation flow where:

1. Marina's reply bubble starts with `"Confirmed! 🎉 Jet Ski Excursion ... Your booking reference is IY398I. Complete your payment here: https://demo.pay/bluemarlin/..."`
2. Above the bubble, a GREEN status banner reads `"Booking confirmed: Jet Ski Excursion, 2026-04-12, 3 guests (Ref: IY398I)"`
3. The customer's next message: `"Ok, sounds good, i will pay, how do you know when i have paid?"`

The customer is confused — and rightly so. The booking is NOT confirmed. It's a soft hold on the calendar with a payment link attached. Marina says "Confirmed!" and the dashboard shows a green "Booking confirmed" tag BEFORE any payment has been received. This is a trust/clarity bug: we're lying to the customer about the state of their booking.

Benson's rule: the word "Confirmed" must not appear until payment has actually come through. While the payment link is still outstanding, the state is "hold placed — awaiting payment" and Marina's wording + the dashboard tag must reflect that.

**Two surface areas** are both saying "confirmed" prematurely:

- **Marina's reply text** (customer-facing, via WhatsApp): The reply bubble in Image #74 starts with "Confirmed! 🎉". This is generated by Marina herself, guided by the WRITING STYLE example at `marina_agent.py:298-299` which shows `"You're all set! Your booking reference is [BOOKING_REF]. Here's your payment link: [PAYMENT_LINK]. See you Saturday! 🎉"`. "You're all set!" implies done. Marina is following the example.
- **Dashboard system banner** (operator-facing): `social_agent.py:716-717` unconditionally writes the system message `f"Booking confirmed: {service_name}, {date}, {guests} guests (Ref: {booking_ref})"` to `wa_messages` on every hold-created path, regardless of payment state. The frontend at `Messages.tsx:359` does `const isBooking = /booking confirmed/i.test(msg.text);` and renders the bubble with a green `CheckCircle2` icon and emerald colors (lines 362-364) when the regex matches.

Both fixes are tiny in isolation but must land together, because fixing only one leaves the contradiction visible (Marina says "held" but dashboard says "confirmed", or vice versa).

### Client-specific wording

This is not a blanket "never say confirmed" — it's state-dependent. Two client types:

- **BlueMarlin** (`payment.timing = "upfront"`): customer is expected to pay via Stripe/demo.pay. Until the payment webhook fires, the state is "hold placed — awaiting payment". Marina should say "held" / "awaiting payment" language. Dashboard shows amber.
- **Adamus** (`payment.timing = "none"`): restaurant reservation, no payment collected upfront. The hold IS the confirmation. Marina should continue to say "confirmed" / "your reservation is set". Dashboard continues to show green.
- **"deposit" timing** (hypothetical third option): treat like "upfront" — payment is still pending when the hold is placed.

`payment.timing` is already in `client.json` (verified: BlueMarlin=`upfront` with `hold_duration_hours=6`, Adamus=`none` with `hold_duration_hours=4`) and already visible to Marina via the auto-generated `=== PAYMENT ===` section in `_build_client_context()` at `marina_agent.py:63-94`. No new config is needed.

### Why this is Brief 163 and not part of Brief 168 (payment hold state machine)

Brief 168 will introduce the full state machine: reaper worker, Stripe link deactivation, late-payment handling, reminder timing. That's several hours of work and involves a background process. Brief 163 is the wording-only fix that kills the most visible/confusing symptom immediately, so the demo stops lying to customers while the bigger state machine work is in flight. The two briefs compose cleanly: when Brief 168 lands, the green "Booking confirmed" tag will finally fire (via a new "payment_received" system message written by the Stripe webhook handler), and Brief 163's amber "Hold placed" tag will be the correct intermediate state until then.

## Why This Approach

**Considered and rejected — just rewrite social_agent.py:716 without touching the prompt.** This would fix the dashboard tag but leave Marina's reply bubble still saying "Confirmed! 🎉". Customer confusion stays. Rejected.

**Considered and rejected — just update the prompt writing style examples without touching social_agent.py.** This would fix Marina's reply but leave the dashboard banner green. Operator confusion stays. Rejected.

**Considered and rejected — add a new `bookings.payment_state` column and derive the dashboard tag color from that.** Cleaner long-term but requires a schema migration, a DB read in the frontend API endpoint, and a JOIN on every conversation load. Overkill for a wording fix. Deferred to Brief 168 where the state machine work justifies the schema change. For now, text-in-the-system-message is the cheapest mechanism and the frontend already matches on text.

**Considered and rejected — inject payment_timing into the system prompt via an explicit variable instead of relying on `=== PAYMENT ===`.** The payment section is already auto-injected by `_build_client_context()` which iterates `client.json` and serialises every top-level key. Marina already sees it. Adding a second injection would be duplicative.

**Chosen approach — match the existing Brief 161 precedent for state-driven wording.** Brief 161 deleted the hardcoded `_build_booking_summary` f-string templates and instead added a BOOKING VALIDATION block to the prompt that tells Marina the wording rules. This brief follows the same pattern: add a CONFIRMATION WORDING rule near BOOKING VALIDATION that reads the payment timing from the client context and switches wording accordingly. The writing style examples (lines 262 and 298-299) are updated to match the new rule. The system message in social_agent.py branches on `_payment_timing`. The frontend gets a third regex class ("hold placed") with amber styling.

Tradeoff carried: the dashboard now has THREE icon states instead of two (escalation=amber triangle, hold=amber clock, confirmed=green check). Slightly more visual vocabulary for the operator to learn, but that's the whole point — the distinction between "held" and "confirmed" is real and has been invisible until now.
