# EXPLANATION 223 — Backend taskNumber for /tasks

## In one sentence

Every task now has a stable TASK-### number assigned by the backend, so when Calvin says "look at TASK-022," Benson sees the same TASK-022 in his browser instead of seeing it labeled TASK-014.

## What's changing and why

Until today, the little TASK-### badge that appears on every task card on the dashboard was generated by the operator's browser. Each browser kept its own private counter in local storage and assigned numbers in the order it happened to load tasks. Two consequences: clearing browser data reset every number, and two different browsers (or two different operators) saw two different numbers for the exact same task. That made it impossible to reliably refer to a task by its number out loud or in a message.

Now the number is allocated and stored on the server when a task is created. Every browser that fetches that task gets the same number, and the number never changes — not when the task moves to done, not when it's reopened, not when it's edited. The frontend was already written to defer to a backend-supplied number when one exists, so the moment this lands, the browser-side counter quietly stops mattering and the badges become consistent across machines.

## Step by step — what the code does now

STEP: New column on the tasks table

The first time the system opens its database after this change, it adds a new "task number" column to the tasks table. The step is wrapped so that running it again does nothing — once the column exists, it just moves on. This means the change is safe to deploy on top of an existing database without any manual migration.

STEP: One-time backfill of existing tasks

Right after adding the column, the system looks for any existing tasks that don't yet have a number. It sorts them oldest-first (by when they were created) and assigns numbers starting from 1: the oldest task becomes TASK-001, the next becomes TASK-002, and so on. Because new connections also run this check, on every later connection the backfill query returns nothing and the work is skipped.

STEP: Creating a new task

When an operator creates a new task, the system asks the database for the highest number currently in use, adds 1, and writes that number into the new task's row at the same time as the rest of the task data. This means the number is assigned exactly once, at creation, and is durable — it lives in the database, not in any browser.

STEP: Reading a task

When the system loads a task, it now also reads the task's number along with everything else. The task number travels with the task data wherever it goes inside the backend.

STEP: Sending tasks to the frontend

When the dashboard's tasks endpoint formats a task for SR's frontend, it includes the task number under the field name "taskNumber" (the camelCase shape the frontend expects). The frontend's existing code recognizes this field and uses the backend's number directly instead of allocating a new one in browser storage.

## Edge cases

- If an operator had been seeing a task as TASK-014 in their browser before this change, that same task may show up as a different number after the change — for example TASK-007. The new numbers come from creation order in the database, which doesn't necessarily match the order each browser happened to load tasks in. This is a one-time visual hiccup. From that point forward the number is stable everywhere.

- If a task is deleted, its number is not reused. The next created task gets one higher than the current maximum, so gaps can appear in the sequence (e.g., TASK-007 might be missing if task 7 was deleted). This is intentional — reusing numbers would break "the number never changes" for anyone who already referenced the deleted number.

- The backfill query (the SELECT that looks for tasks without a number) runs every time the database is opened, even after every existing task already has a number. After the first run it finds zero rows and does nothing, so the cost is one cheap database query per connection. If this ever becomes a performance issue on a large tasks table, it can be moved behind a per-process flag — but the tasks table is small (operator tasks, not customer messages) and this matches the pattern used by previous schema changes in the same place.

- If two operators create a new task at the exact same instant on the same workspace, both creations read the current maximum and add 1, so in theory both could try to use the same next number. In practice the database serializes writes for this table, and the tasks table is small enough that this hasn't been a contention point. If it ever became one, swapping in a dedicated counter table would resolve it.

- "Per workspace" lines up with how each tenant runs in its own container with its own database, so numbers are scoped automatically — BlueMarlin's TASK-001 and Adamus's TASK-001 are different tasks, not a collision.

## What did NOT change

This change only touches the tasks table and the tasks endpoint that powers SR's Tasks page on the dashboard. Marina's prompt, the customer conversation flow, the booking handling, the WhatsApp and DM webhooks, and the email pipeline are all untouched. No customer-facing message wording changes. Operators won't see anything different on the conversation side — only the TASK-### badges on the Tasks page become consistent across browsers and operators.
