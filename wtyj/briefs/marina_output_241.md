# OUTPUT 241 — Appointment alerts using shared alert destinations

## What was done

Added appointment alerts on top of Brief 217/226/240's alert plumbing. Per-step shipped:

1. **Schema (state_registry.py):** 4 ALTER ADD COLUMN — `alert_deliveries.alert_type TEXT NOT NULL DEFAULT 'escalation'` (existing rows retro-labeled correctly), `alert_deliveries.appointment_id INTEGER` (nullable), `alert_settings.alert_type_escalation_enabled INTEGER NOT NULL DEFAULT 1`, `alert_settings.alert_type_appointment_enabled INTEGER NOT NULL DEFAULT 1`. Idempotent try/except OperationalError pattern.
2. **Generalized `record_alert_delivery`** with two optional kwargs at the END (`alert_type="escalation"`, `appointment_id=None`) — all 8+ existing positional callers in `_fire_escalation_alerts` keep working without changes.
3. **New helper `appointment_alert_already_sent(appointment_id, channel, destination)`** — queries alert_deliveries for an existing row with `alert_type='appointment'` AND matching tuple AND `status IN ('sent', 'failed')`. `'skipped'` rows do NOT count (intentional — skipped means "we couldn't send" and should retry).
4. **New dispatcher pointer + setter** in state_registry: `_appointment_alert_dispatcher` global + `set_appointment_alert_dispatcher(fn)` mirrors Brief 217's `_alert_dispatcher` pattern. Decouples state_registry from importing dashboard.
5. **`appointment_upsert` rewritten** to detect transition-to-confirmed (insert+confirmed OR update from non-confirmed→confirmed). Re-saves of `confirmed` status do NOT fire (layer-1 dedup via transition detection). Best-effort dispatcher call wrapped in try/except so a dispatcher failure NEVER blocks the row write. Per-tenant gate: reads `alertTypes.appointments` from `get_alert_settings` before firing.
6. **New `_fire_appointment_alerts` + `_build_appointment_subject` + `_build_appointment_body`** in dashboard/api.py. Subject: `"Appointment confirmed: {name} — {time}"` (falls back to name only if no time). Body: Customer / Channel / Topic / Time / Location / dashboard CTA. Per-channel layer-2 dedup via `appointment_alert_already_sent` BEFORE each send. WhatsApp branch reuses Brief 240's `get_resolved_operator_whatsapp_route` + `send_dm_reply` — no Meta fallback. Telegram/Messenger record `skipped` with reason. Each delivery passes `alert_type="appointment"`, `appointment_id=<id>`, escalation_id=None.
7. **`_fire_escalation_alerts` short-circuit:** added an `alertTypes.escalations` gate at the top of the dispatcher (before reading channels_cfg) — when False, return immediately. Default True for backward compat.
8. **Extended `get_alert_settings`** to read the 2 new alert_type_*_enabled columns and surface them as a top-level `alertTypes: {escalations: bool, appointments: bool}` block. Synthesized-default branch and real-row branch both updated. **Extended `save_alert_settings`** with optional `alert_types` kwarg (defaults to both True); persists into the 2 new columns via the same `ON CONFLICT DO UPDATE` Brief 240 introduced — bootstrap-only `whatsapp_zernio_*` columns from Brief 240 still survive a Settings save.
9. **Pydantic model + endpoint:** new `AlertTypesConfig` model (escalations, appointments — both default True); `AlertSettingsRequest` extended with `alertTypes: AlertTypesConfig = AlertTypesConfig()` (default supplied so old frontend payloads without the field still parse cleanly); `PUT /settings/escalation-alerts` passes `req.alertTypes.model_dump()` to `save_alert_settings`. The GET endpoint required no signature change since `get_alert_settings` was extended in Step 8.
10. **6 new tests** appended to `wtyj/tests/social/test_217_alert_delivery.py` + a `_wipe_appointments_for(conversation_id)` helper (parallel to Brief 240's `_wipe_escalations_for`).

## Reachability gap (documented in brief, not surprise)

No production code path currently sets `appointments.status = 'confirmed'`. The dispatcher this brief installs is wired but dormant in production until a future caller flips an appointment to confirmed (operator dashboard endpoint to confirm OR Marina-side detection of "yes Friday 12:00 works"). Issue #4 explicitly asked for the alert wiring, not the upstream confirmation signal — those are separate product decisions reserved for a follow-up brief. Tests exercise the trigger by calling `appointment_upsert(..., status='confirmed', ...)` directly, which is a real behavioral assertion of the wired path.
