## Context

SR rebranded the public face of the platform from `wetakeyourjob.com` to `unboks.org` (`unboks.com` does not exist). New canonical API hostname going forward is **`api.unboks.org`**. Customer-facing site lives at `unboks.org`, operator dashboard at `dashboard.unboks.org`, public contact email is `hello@unboks.org` (Google Workspace). The internal codebase and `wetakeyourjob.com` infrastructure stay alive — only the public domain moves.

While we shipped Brief 199, **Replit's AI accidentally generated a parallel Node/Express/Postgres backend** in SR's `unboks-dashboard-api` Replit project (repo `calvin835/unboks-dashboard-api`). DNS for `api.unboks.org` was pointed at that Replit deploy. The fake Node backend has its own (empty) Postgres tables, a stubbed Zernio webhook, and a stubbed escalation reply route — **none of it is connected to anything**. Meanwhile our real Python backend (`wtyj-agent` at `api.wetakeyourjob.com`) has been receiving live traffic the whole time:

```
ssh root@108.61.192.52 "docker exec wtyj-unboks python3 -c 'import sqlite3; c = sqlite3.connect(\"/app/data/state_registry.db\"); print(\"whatsapp_threads:\", c.execute(\"SELECT COUNT(*) FROM whatsapp_threads\").fetchone()[0])'"
→ whatsapp_threads: 90
```

90 real WhatsApp conversations are in the unboks tenant DB. calvin-csa was actively replying to messages on Calvin's promo number throughout. The dashboard at `dashboard.unboks.org` simply can't see them because it calls `api.unboks.org` (the empty Replit Node) instead of `api.wetakeyourjob.com` (our Python).

SR's audit (paraphrased): "WhatsApp ingestion is missing — please build it in the Node backend." His proposed fix would mean rebuilding from scratch — duplicating Marina, dm_agent, customer files, escalation rules, multi-tenant routing, the canary deploy queue, the 907-test regression suite, etc. — in Node. Months of throwaway work to land at functional parity with what already exists.

The actual fix is **DNS-level + nginx-level**: re-point `api.unboks.org` at our VPS (`108.61.192.52`), add an nginx server block that handles SR's `/api/{client}/...` URL shape, and the dashboard reconnects automatically. No code change in his Replit project, no code change in our Python codebase. The fake Node backend becomes orphaned and can be archived later.

This brief covers our half of the cutover — pre-position the nginx config + smoke-test scripts NOW, so the moment SR changes DNS, the cutover takes ~5 minutes (certbot + reload). DNS change is SR's hand to play; we don't action it here.

### URL-shape mismatch — SR's frontend vs our backend

SR's frontend (auto-generated by Replit's AI to match the auto-generated Node backend) calls these paths:

| What the dashboard calls | What our backend serves at |
|---|---|
| `POST /api/unboks/dashboard/api/login` | `POST /unboks/dashboard/api/login` |
| `GET /api/unboks/dashboard/api/messages/conversations` | `GET /unboks/dashboard/api/messages/conversations` |
| `GET /api/unboks/dashboard/api/escalations` | `GET /unboks/dashboard/api/escalations` |
| `GET /api/healthz` | `GET /health` (per Brief 192) |

Our backend's existing nginx at `api.wetakeyourjob.com` strips the tenant prefix only (`/unboks/foo` → `/foo` on container `:8004`). The new `api.unboks.org` server block needs to strip BOTH the `/api/` AND `/{tenant}/` prefixes (`/api/unboks/foo` → `/foo` on container `:8004`). Single-line nginx pattern: `proxy_pass http://127.0.0.1:8004/;` — the trailing slash on `proxy_pass` instructs nginx to replace the matched `location /api/unboks/` prefix with `/`.

For `/api/healthz` (single global route on SR's frontend, called on dashboard mount), proxy to one tenant's `/health` endpoint. BlueMarlin (`:8001`) is the obvious choice since it's the canary. If BlueMarlin is down, healthz returns 502 — that's correct, the platform IS unhealthy.

### Why no Python source change

- Backend already serves the right paths (`/{tenant}/dashboard/api/...`) from Brief 175.
- CORS already allows `unboks.org` origins (commit `c994037` in Brief 199 follow-up).
- All 4 tenant containers are already running on `:8001`/`:8002`/`:8003`/`:8004`.
- Only nginx needs to learn about the new hostname and the `/api/` prefix shape.

This is fundamentally a **VPS infrastructure brief**: nginx config, TLS cert, smoke tests. No `python3 -m pytest wtyj/tests/` deltas. The regression baseline of **907 passing** stays at 907 (no Python tests added).

## Why This Approach

**Considered alternatives:**

1. **Build the Node backend SR's audit asked for.** Rejected: rebuilds known-working Python infrastructure in a different language, throws away 4 tenants of production data + the canary pipeline, takes weeks. The Node backend is empty — there's nothing to migrate.

2. **Leave api.unboks.org pointing at Replit and migrate dashboard.unboks.org to call api.wetakeyourjob.com directly.** Rejected: requires SR to edit his frontend's BASE_URL, and every future client onboarded would need the same DNS-level fix anyway. Better to do it once, properly, now. Also: `api.unboks.org` is the canonical brand-aligned hostname — that should be the API URL going forward, not `api.wetakeyourjob.com`.

3. **Run BOTH backends in parallel and sync data.** Rejected: dual sources of truth, 2x infrastructure to maintain, no benefit. Violates Rule 1 (one Claude call per inbound message) implicitly — two separate processing pipelines.

**Chosen approach:** DNS re-point + nginx server block on our VPS. SR changes one DNS record; we add one nginx file + run certbot. Dashboard reconnects automatically with zero code change in either repo. Zero customer-visible downtime (DNS propagation is the only delay, and during propagation the dashboard simply continues hitting the empty Node backend until it resolves to us — no error, just empty data).

**Tradeoff:** The fake Node Replit project lingers as an orphan until manually archived. That's fine — it costs nothing to leave and SR can clean it up at his convenience.

**Tradeoff:** `api.wetakeyourjob.com` continues to serve the same backend. Both hostnames will work simultaneously until we explicitly retire the old one (later brief). This is intentional — it lets us verify the cutover before retiring the old domain, and keeps existing Zernio webhook URLs (`https://api.wetakeyourjob.com/{tenant}/webhooks/zernio`) working without immediate reconfiguration. Migrating Zernio's webhook URLs is out of scope here; works fine where it is.

### Phase A — pre-position (executable now, no DNS change required)

**A1. Create the new nginx site config** at `/etc/nginx/sites-available/api-unboks` on the VPS. Separate file (not appended to `api-wetakeyourjob`) for clean separation and easier rollback. Initial version is HTTP-only (no TLS yet — certbot adds the TLS lines in Phase B). Contents:

```nginx
# api.unboks.org — Layer 1 cutover (Brief 200).
# DNS for api.unboks.org → 108.61.192.52 must be live before this serves real traffic.
# TLS lines added by certbot in Phase B. Until then, this block is HTTP-only.
#
# Request shape: /api/{tenant}/{rest} → forwarded to tenant container as /{rest}
# Both /api/ AND /{tenant}/ prefixes are stripped via the trailing slash on proxy_pass.

server {
    server_name api.unboks.org;

    # Global health route (SR's frontend calls /api/healthz on dashboard mount).
    # Proxies to BlueMarlin's /health — if the canary is down, the platform is down.
    location = /api/healthz {
        proxy_pass http://127.0.0.1:8001/health;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    # Per-tenant routing. /api/{tenant}/foo → container's /foo
    # Trailing slash on proxy_pass replaces the entire matched prefix.

    location /api/staging/ {
        proxy_pass http://127.0.0.1:9001/;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    location /api/bluemarlin/ {
        proxy_pass http://127.0.0.1:8001/;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    location /api/adamus/ {
        proxy_pass http://127.0.0.1:8002/;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    location /api/consultadespertares/ {
        proxy_pass http://127.0.0.1:8003/;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    location /api/unboks/ {
        proxy_pass http://127.0.0.1:8004/;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    # Catch-all for unknown paths — return 404 instead of accidentally
    # proxying to BlueMarlin (which is what api-wetakeyourjob.com does for backward compat).
    # api.unboks.org is the new canonical host — strict 404 on unknown routes.
    location / {
        return 404;
    }

    listen 80;
}
```

# Brief 200 — cutover script for api.unboks.org TLS issuance.
# Prerequisite: DNS A record for api.unboks.org points at 108.61.192.52.
# Run on VPS as root: bash wtyj/scripts/cutover_unboks_domain.sh
set -e

DOMAIN="api.unboks.org"

RESOLVED=$(dig +short "$DOMAIN" | tail -1)

echo ""
echo "✓ Cutover complete. Verify externally:"
echo "  curl -s https://$DOMAIN/api/healthz"

## Tests

This brief touches **no Python source code**. The pytest regression baseline of **907 passing** stays at 907. There are no new pytest files.

Behavioral verification happens via three executable scripts:

1. **`nginx -t`** (Phase A, step A3) — validates the new server block parses correctly and doesn't conflict with existing config. Run on VPS during execution.

2. **Phase A smoke (step A6)** — Host-header-driven curls against localhost confirm the new server block routes correctly to all 4 tenant containers + the global healthz, AND that unknown paths return 404. This proves the routing logic works BEFORE DNS is flipped, so we know Phase B will work the moment DNS resolves.

3. **`wtyj/scripts/smoke_unboks_domain.sh`** (Phase B, step B4) — 6 external checks against `https://api.unboks.org`: healthz, 3 tenant health endpoints, login returns JWT, conversations endpoint returns 200 with the JWT. Exits non-zero on first failure.

Why no pytest tests:
- Source-level string guards (`assert "location /api/" in open("/etc/nginx/...").read()`) are tautologies banned by CLAUDE.md.
- Nginx config has no Python interface to mock-test.
- A pytest test that shells out to `nginx -t` adds nothing over the direct VPS check in step A3.
- The behavior we care about (nginx routes correctly) is tested at the only place it can be tested — against a live nginx with the config loaded.

This is consistent with how Briefs 195/196 handled CI/CD changes — the test surface is shell scripts, not Python.

## Rollback

**Phase A rollback** (if `nginx -t` fails or the new server block accidentally affects `api.wetakeyourjob.com`):
```bash
ssh root@108.61.192.52 "rm /etc/nginx/sites-enabled/api-unboks && nginx -t && systemctl reload nginx"
```
This removes the new site without deleting the file (`sites-available/` keeps the config for re-enabling). `api.wetakeyourjob.com` is unaffected — its config is in a separate file and was never touched.

**Phase B rollback** (if certbot fails or post-cutover smoke tests fail):
```bash
ssh root@108.61.192.52 "rm /etc/nginx/sites-enabled/api-unboks && nginx -t && systemctl reload nginx"
# Then ask SR to revert DNS A record for api.unboks.org back to the Replit IP.
```
The dashboard.unboks.org frontend will go back to hitting the empty Replit Node backend (showing no WhatsApp data) — same broken-but-visibly-fine state as before this brief. No data loss, no customer impact (we have no paying customers on dashboard.unboks.org yet).

**Cert rollback** (Let's Encrypt issued cert successfully but we want to revoke):
```bash
ssh root@108.61.192.52 "certbot revoke --cert-path /etc/letsencrypt/live/api.unboks.org/cert.pem --non-interactive"
ssh root@108.61.192.52 "certbot delete --cert-name api.unboks.org --non-interactive"
```
Rare scenario — only needed if cert was issued in error. Normal Phase B rollback above is sufficient otherwise.

The whole brief is fully reversible with one command. There's no DB migration, no schema change, no data write — just nginx config and a TLS cert.
