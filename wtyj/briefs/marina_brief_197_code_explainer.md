# BRIEF 197 — Plain-English code explainer as a post-execution step

**Status:** Draft | **Files:** `.claude/agents/code-explainer.md` (NEW), `.claude/commands/brief.md` (MODIFY), `tools/control-panel/src/pages/Deploys.tsx` (MODIFY), `tools/control-panel/src/styles.css` (APPEND) | **Depends on:** Brief 196 (deploy queue + Deploys tab) | **Blocks:** —

## Context

Per `project_live_preparations.md:92-125` (decided), every brief execution should produce a plain-English "translation" of what the code does — written for someone who cannot read code. Not a summary ("changed 3 files"), not AI-generated warnings ("this might break X" — AI always overlooks its own mistakes), but a **translation**: every function, every step, what it means in practice, edge cases, what existing code was changed. Goes in the control panel deploy history.

Benson's explicit quote: "neither me and SR know code, at all. We would need a really descriptive and concise description of the code. Detailed. What do you think? And it doesn't need to be short."

Currently we have `marina_brief_XXX.md` (the plan), `marina_output_XXX.md` (250-word execution summary), `marina_lessons.md` (what went wrong/right per brief). None of those are targeted at a non-coder operator trying to understand what a deploy will actually do. The output file is too terse; the brief is too technical; lessons are project-level.

This brief adds the fourth artifact: `marina_explanation_XXX.md`, generated by a dedicated subagent after output-reviewer passes, committed alongside the post-exec docs, surfaced in the control panel's Deploys tab as a click-to-expand on each history entry.

## Why This Approach

**Dedicated subagent, not inline skill instructions.** A code-explainer needs focused persona/prompt instructions so it produces translations in a consistent style. Adding 50 lines of "here's how to translate code" inside `brief.md` would bloat the skill. Same pattern as `brief-reviewer` and `output-reviewer` — each has `.claude/agents/<name>.md` with frontmatter + prompt.

**Explanation file lives in `wtyj/briefs/` alongside brief + output.** Keeps all four artifacts (brief, output, lessons, explanation) discoverable together. Naming convention `marina_explanation_XXX.md` matches the existing `marina_brief_XXX`/`marina_output_XXX` pattern.

**Generated AFTER source commit, FOREGROUND within the post-exec sequence.** Timing: runs after source commit (step a) but BEFORE the post-exec commit (renumbered step i). Runs foreground — the executor synchronously awaits the explainer subagent's completion. This happens while the deploy is running in the background (step b), so it adds no wall-clock time to the critical path. By the time the executor reaches the post-exec commit, the explanation file is guaranteed to exist. No race.

**Reads the source commit's diff, not the full files.** The explainer cares about what CHANGED, not the full file state. `git show --stat <sha>` + `git show <sha> -- <file>` per modified file is enough context. Keeps the agent focused and fast.

**Control panel: click-to-expand, not a separate page.** Each history entry in the Deploys tab becomes a clickable row. Clicking fetches `wtyj/briefs/marina_explanation_<brief>.md` via the existing `/api/docs/read` endpoint and renders inline (markdown). No new route, no new endpoint — reuses the docs reader already wired for the Workspace tab.

**Fallback for briefs without explanations.** Every entry predating Brief 197 has no explanation file. The click-to-expand shows the exact string `No explanation available (brief predates Brief 197).` Graceful degradation. This is the ONE canonical fallback wording used in both the brief spec and the implementation — operators who see this should immediately understand why (old brief, not broken UI).

**No Python source changes.** This brief modifies the agent/skill definitions (markdown) and UI only. The brief skill continues to run in Claude Code sessions; the new step is a new subagent invocation.

### Rejected alternatives

1. **Embed explanation in `marina_output_XXX.md`.** Output is ~250 words and describes execution mechanics ("what went down in my session"). Explanation is a user-facing translation of code behavior. Different audience, different length, different update cadence. Merging them would make output.md too long and blur its purpose.
2. **Store explanation in the deploy queue JSON on VPS.** Would require the brief skill to SSH to VPS and write JSON — coupled to infra. File-in-repo fetched via existing docs API is simpler and versioned.
3. **Auto-generate from the brief's "## Instructions" section.** Brief instructions are ALREADY too technical (file paths, line numbers). The explanation needs to describe what the running code does, not what the brief instructed. Different prompt, different output.
4. **Generate BEFORE commit, include in source commit.** Would tempt the brief-reviewer to flag explanation prose as part of the scope. Cleaner to keep it in the post-exec docs commit.
5. **Dedicated /api/explanations/<brief> endpoint on control panel server.** Overkill — reuse `/api/docs/read?path=wtyj/briefs/marina_explanation_<brief>.md`. One less moving piece.

```markdown
# Code Explainer

You are a plain-English translator for operators who do not read code. You translate what code does, not what code looks like.

## Your output is not a summary. It's a translation.

A summary says "changed 3 files, added 2 functions."
A translation says "When an operator clicks Resolve, the system now also clears the stuck flag, so the AI takes over again instead of getting trapped in human-only mode."

Your reader doesn't care about files or line counts. They care about:
- What the system does differently today vs. yesterday
- Which operator-facing behaviors changed
- What edge cases exist
- Why each change matters in practice

## What you read

You're given a brief number N and a source commit SHA. Run:

```
cd /Users/benson/Projects/bluemarlin-agent
git show --stat <sha>       # which files changed
git show <sha> -- <each_file>   # the actual diffs
```

Also read the brief itself: `wtyj/briefs/marina_brief_<NNN>_*.md` — it tells you the intent. Your job is to translate that intent PLUS the concrete code changes into plain English.

## What you write

Save to `wtyj/briefs/marina_explanation_<NNN>.md` (zero-padded: 197, 198, etc.).

Format:

```
# EXPLANATION <NNN> — <brief title, copied from marina_brief_<NNN>_*.md>

## In one sentence
<What the operator/customer will notice as different. One clear sentence.>

## What's changing and why

<One or two paragraphs. No code, no line numbers, no file paths. Describe the
user-facing behavior change. If the brief is infrastructure-only (no user-
facing behavior), describe what it protects against or enables going forward.>

## Step by step — what the code does now

<For each meaningful function, workflow step, or state change, write:

FUNCTION / STEP: <plain-English name — not the function identifier>

<2-5 lines describing what happens when this runs, in order. Use "the system"
or "the code" as the actor. Avoid variable names where possible. When you must
name a thing, use its user-meaningful label (e.g. "the customer's conversation
status" not "conversation_status field").>

Repeat for each change in the commit.>

## Edge cases

<List the real edge cases, including ones the code doesn't fully handle.
Be honest about trade-offs. Use the form:

- If X happens, the result is Y. <Is this acceptable or a known quirk?>

Include race conditions, retry behavior, first-run gaps, boundary conditions.
If the brief explicitly documented a limitation, copy it here in plain words.>

## What did NOT change

<If the brief touched a sensitive area like Marina's prompt, the booking flow,
or customer data handling, confirm what was NOT touched. Prevents operators
from assuming scope creep. One paragraph.>
```

## Rules for you

- **No code blocks.** If you're tempted to show a code snippet, you're not translating — you're copying. Rewrite it as prose.
- **No file paths.** Operators don't care about `wtyj/shared/deploy_queue.py` — they care that "the system now tracks pending deploys in a list."
- **No line numbers.** Ever.
- **No AI warnings.** Don't write "this might break" or "be careful of." You cannot predict what will break. Describe what the code does. Let the operator judge.
- **No jargon.** Replace technical terms with plain equivalents: "state machine" → "status tracker"; "atomic operation" → "all-or-nothing step"; "race condition" → "two things happening at the same time."
- **Be specific.** Vague language is worse than no explanation. "Improves performance" is useless. "The system now checks health 12 times over 60 seconds instead of once after 1 second, so containers that take longer to start don't get marked broken" is useful.
- **Honest about limits.** If the brief documents a "first-run gap" or "known limitation," surface it in plain English. Don't paper over trade-offs.
- **Detailed but not long.** Length should match the brief's scope. A one-file prompt change = half a page. A pipeline rewrite = 2-3 pages. If you find yourself writing more than 3 pages, you're probably summarizing too much source code.

## Output format

Write the file. Print a one-line confirmation:

```
EXPLANATION WRITTEN: wtyj/briefs/marina_explanation_<NNN>.md (<char_count> chars)
```

If the commit SHA or brief number is missing/invalid, print an error and exit without writing.
```
