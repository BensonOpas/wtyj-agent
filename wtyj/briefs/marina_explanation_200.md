# EXPLANATION 200 — api.unboks.org cutover (Layer 1 of wtyj→unboks rebrand)

## In one sentence
The VPS now answers to a second public name — `api.unboks.org` — using the new URL shape the Unboks dashboard expects, so the moment the dashboard's domain points at our server it will start showing real WhatsApp conversations instead of an empty page.

## What's changing and why

The platform is being rebranded from "wetakeyourjob.com" to "unboks.org". The customer-facing dashboard at `dashboard.unboks.org` was wired up to call `api.unboks.org`, but that hostname was pointing at an unrelated empty test backend that someone had set up on Replit. Meanwhile, the real Python backend at `api.wetakeyourjob.com` had been quietly receiving live traffic the whole time — there are already about 90 real WhatsApp conversations sitting in the Unboks tenant database, generated by the customer service operator replying through Calvin's promo number. The dashboard simply couldn't see them because it was asking the wrong server.

This change pre-positions everything our side of the fence needs so that, when the domain owner finally points `api.unboks.org` at our VPS, the dashboard will start working in roughly five minutes — the time it takes to issue an HTTPS certificate and reload the web server. There is no Python code change, no database change, and no behavior change for any customer or tenant today. The existing `api.wetakeyourjob.com` keeps working exactly as it did. Once the domain flip happens later, both names will serve the same backend at the same time.

The tricky part the new web server config solves is that the Unboks dashboard's URLs are shaped differently than the old hostname's URLs. The dashboard sends requests like `/api/unboks/dashboard/api/messages/conversations`. The backend listens for `/unboks/dashboard/api/messages/conversations` — no `/api/` in front. The new server config strips the `/api/` AND the tenant name before handing the request to the right tenant container, so both URL shapes serve the same backend without any code knowing the difference.

## Step by step — what the code does now

NEW PUBLIC HOSTNAME ON THE WEB SERVER

A new server block was added on the VPS that listens for requests addressed to `api.unboks.org`. It only activates when a request explicitly names that hostname — until DNS is flipped, no real traffic hits it, so it sits idle and harmless. The existing hostname's config file was not touched.

ROUTING REQUESTS TO THE RIGHT TENANT

When a request arrives for `api.unboks.org`, the new server block looks at the path. If the path starts with `/api/bluemarlin/`, the request is forwarded to the BlueMarlin tenant container with the `/api/bluemarlin/` chopped off the front. The same trick handles `/api/adamus/`, `/api/consultadespertares/`, `/api/unboks/`, and `/api/staging/` — each is rewritten to look like a normal request and handed to the matching tenant container on its private port. So a dashboard call for `/api/unboks/dashboard/api/messages/conversations` reaches the Unboks container as `/dashboard/api/messages/conversations`, which is exactly what the backend already serves.

GLOBAL HEALTH CHECK

The Unboks dashboard pings a single global URL — `/api/healthz` — every time it loads, to confirm the platform is alive. Since "the platform" doesn't have one health endpoint (each tenant has its own), this one route is wired to BlueMarlin's health check. BlueMarlin is the canary — if the canary is broken, the platform is broken — so its status is a fair stand-in for the whole system.

UNKNOWN PATHS RETURN 404

Any path that isn't a known tenant route or the global health route returns a clean "not found" response. The old hostname forwards unknown paths to BlueMarlin for backward compatibility, but the new hostname is strict — it only answers what it's been told to answer.

POST-CUTOVER SMOKE TEST SCRIPT

A new shell script was added to the repo that runs six checks against the live `https://api.unboks.org` once it's pointed at us. It hits the global health route, the BlueMarlin / Adamus / Unboks tenant health endpoints, logs into the Unboks dashboard with the operator password to confirm a session token comes back, then uses that token to fetch the conversations list. If any check fails, the script stops and exits with an error. This is the script someone runs by hand right after the domain is switched, to confirm the cutover succeeded.

CUTOVER SCRIPT FOR THE DOMAIN FLIP

A second new shell script does the actual switch-over work on the VPS. It first checks that the domain's DNS record actually resolves to our server's address — if the domain still points somewhere else, it bails out with a clear message instead of doing anything destructive. If DNS is good, it asks Let's Encrypt for a fresh HTTPS certificate, validates the web server config, and reloads the web server. The script is not run yet — it sits in the repo waiting for the day the domain is officially handed over.

## Edge cases

- If someone visits `https://api.unboks.org` today before DNS is changed, they hit the wrong (empty) backend, exactly as before. Our new config does nothing because no traffic reaches it. That's expected — Phase A is positioning, not the flip itself.

- If DNS is changed but the cutover script hasn't been run yet, browsers will hit our server over plain HTTP and likely fail to load (the dashboard speaks HTTPS). The window between DNS flip and certificate issuance should be minutes. Anyone refreshing during that window sees a broken connection, not bad data.

- If the certificate request fails — usually because DNS hasn't fully spread to Let's Encrypt's validation servers — the cutover script stops before reloading the web server. Nothing is left in a half-applied state.

- If BlueMarlin happens to be down the moment the dashboard loads, the global health check will return a server error, which is the correct signal — the platform really isn't healthy at that moment.

- During DNS propagation (which can take minutes to hours depending on caches around the internet), some users will still be routed to the empty Replit backend and see no conversations. There is no error message — just an empty inbox. This is unavoidable with any DNS change and clears itself once the new address is universally cached.

- Both hostnames will keep working in parallel after the cutover. The old `api.wetakeyourjob.com` is intentionally left alive so existing webhook URLs registered with the WhatsApp provider keep working. Nothing forces traffic onto the new name; the dashboard just chooses the new name on its own.

- The empty Replit backend that was previously misconnected to `api.unboks.org` becomes an orphan — nothing points at it after DNS changes, but it is not deleted as part of this work. It costs nothing to leave it sitting.

- The actual web server config file lives only on the VPS, not in this repo. If the server is rebuilt from scratch, the file would need to be recreated by hand or recovered from a snapshot. The brief documents what's in it, but the file itself isn't tracked.

## What did NOT change

No Python source code was modified. The Marina prompt, the booking flow, the message processing pipeline, the canary deploy queue, the tenant container configuration, customer data handling, and the WhatsApp ingestion path are all untouched. The existing `api.wetakeyourjob.com` hostname continues serving every tenant exactly as before. No database migrations, no schema changes, no environment variable changes, no container restarts. The 907-test regression baseline is unchanged because there are no Python changes to test. The only files added are two shell scripts and a brief; the only thing changed on a running system is one new file on the VPS that won't see traffic until a third party flips a DNS record.
