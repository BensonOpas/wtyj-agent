# OUTPUT 240 — Operator WhatsApp alerts via Zernio + delivery-status truth

## What was done

Switched operator WhatsApp escalation alerts from the legacy Meta Cloud API path (which silently dropped messages outside Meta's 24-hour customer service window — the cause of Calvin not receiving alerts) to Zernio's `send_dm_reply`, the same provider as unboks customer chat. Three new ALTER ADD COLUMN statements on `alert_settings` (`whatsapp_zernio_conversation_id`, `whatsapp_zernio_account_id`, `whatsapp_zernio_resolved_at`) capture the bootstrap Zernio route. New `state_registry.set_resolved_operator_whatsapp_route` + `get_resolved_operator_whatsapp_route` helpers. New auto-resolve hook in `_process_zernio_event` (after Brief 238 tenant_guard, before empty-text skip): when an inbound WhatsApp's normalized sender_id matches the configured `whatsapp_destination`, persist the conv_id + account_id automatically. Rewrote the WhatsApp branch of `_fire_escalation_alerts`: read the resolved route via the new helper; on resolved → call `send_dm_reply` with the route's identifiers + the Brief 239 rich body (record `sent`/`failed` with exact provider reason); on unresolved → record `skipped` with reason `zernio_operator_destination_not_resolved` (no fake `sent`, no Meta fallback). Switched `save_alert_settings` from `INSERT OR REPLACE` to `INSERT … ON CONFLICT DO UPDATE` so the bootstrap-only columns survive Settings saves. `GET /settings/escalation-alerts` now returns `channels.whatsapp.zernioResolved: bool` so SR's frontend can render bootstrap-needed copy. **Brief-reviewer:** PASS round 1, zero issues. **Output-reviewer:** TBD.

## Unexpected findings

**Test isolation bug surfaced (existing latent issue, not Brief 240's).** Three Brief 239 tests + the four new Brief 240 tests share the dev-mode SQLite DB (`wtyj/data/state_registry.db`). Without per-test cleanup, a prior run leaves `pending_notifications` rows with the same test cids; on the next run, Brief 239's dedup-update path fires and the test's expected create-from-scratch flow is suppressed. Brief 239 worked the day it shipped because the DB was clean; on Brief 240's second pytest run, three Brief 239 tests started failing. Fix: added a `_wipe_escalations_for(customer_id)` helper to the test file and inserted it at the start of all 7 affected tests. Also removed the obsolete Brief 217 test `test_create_pending_notification_fires_whatsapp_alert_to_configured_destination` whose contract (Meta `send_whatsapp_message` call for operator WA alerts) is replaced by Brief 240's three new WA tests. Both fixes are in scope: the test-file is the same per-source-module file Brief 240 was already extending, and the obsolete test would have continued to fail with the new dispatcher.
