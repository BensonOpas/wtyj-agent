You are stepping back to check your own thinking. This is an anti-tunnel-vision check.
STOP whatever you were about to do and answer these four questions honestly:

1. STATE THE PROBLEM: What are we actually trying to solve? One sentence.

2. DOES SOMETHING ALREADY EXIST? Search the web, check if a library, service, or tool already does this. If yes, use it instead of building. If you haven't searched, search now.

3. WHAT'S THE SIMPLEST APPROACH? Describe the absolute minimum implementation. If what you were about to build is more complex than this, explain why the complexity is justified.

4. AM I SOLVING THE RIGHT PROBLEM? Is the user's request the real issue, or is there a deeper/different problem underneath?

After answering, tell the user what you found. If your current approach is wrong or overcomplicated, say so directly and propose the simpler path.

This skill can be invoked by the user (/scope) or by yourself when you notice you're deep in implementation without having validated the approach.
